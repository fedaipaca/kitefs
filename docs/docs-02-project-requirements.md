## 1. Functional Requirements

> _What the system must DO. Describe capabilities and behaviors. Priority values:_

### 1.1 Feature Group Definition

| ID         | Requirement                                                                                                                                                                                                                                                 | Priority  | Traces To  |
| ---------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------- | ---------- |
| FR-DEF-001 | The user shall be able to define a feature group as code using the Python SDK, specifying: a unique name (`name`), a storage target (`storage_target`), a single `EntityKey` (with name and data type), a single `EventTimestamp` (with name and data type, must be `DATETIME`), and one or more `Feature` instances (each with a name, data type, optional description, and optional expectations via the `Expect` builder). The structural and feature columns are provided as separate, typed constructor parameters (`entity_key`, `event_timestamp`, `features`) rather than a mixed list. | Must Have | G-1, PP-2  |
| FR-DEF-002 | The system shall support the following feature data types: `string`, `integer`, `float`, `datetime`.                                                                                                                                                        | Must Have | G-1        |
| FR-DEF-003 | The user shall be able to specify the storage target for a feature group: `OFFLINE` (offline store only) or `OFFLINE_AND_ONLINE` (offline store and materializable to the online store).                                                                    | Must Have | G-1, PP-3  |
| FR-DEF-004 | The user shall be able to define feature expectations (business-level constraints) on each `Feature` using the `Expect` fluent builder — including `.not_null()`, `.gt(v)`, `.gte(v)`, `.lt(v)`, `.lte(v)`, and `.one_of(v)`. Expectations are not available on `EntityKey` or `EventTimestamp` fields. | Must Have | G-5, PP-7  |
| FR-DEF-005 | The user shall be able to specify the validation mode independently for each validation gate (ingestion, offline retrieval), choosing from `ERROR`, `FILTER`, or `NONE`. The gate list is designed for future expansion (e.g., an online retrieval gate); the MVP implements two gates.                                                                                           | Must Have | G-5, PP-7  |
| FR-DEF-006 | The user shall be able to add metadata to a feature group definition via a typed `Metadata` object with three optional fields: a human-readable description (`str`), owner name (`str`), and tags (`dict[str, str]`). All fields default to `None`, allowing the metadata parameter itself to be omitted entirely. | Must Have | PP-5, PP-6 |
| FR-DEF-007 | The system shall enforce that each feature group has exactly one `EntityKey` and exactly one `EventTimestamp`. This is guaranteed by the constructor signature (`entity_key: EntityKey`, `event_timestamp: EventTimestamp`) — the type system prevents missing or duplicate structural columns at definition time, not at `apply()` time. | Must Have | G-1        |

### 1.2 Feature Registry

| ID         | Requirement                                                                                                                                                                                                                                                                                                                                                              | Priority    | Traces To       |
| ---------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ----------- | --------------- |
| FR-REG-001 | The system shall persist all registered feature group definitions in a feature registry stored as a JSON file within the project's storage root (default: `./feature_store/`). The format shall be deterministic (stable key ordering, consistent formatting) to produce meaningful Git diffs. The registry file shall be Git-versionable. Each feature group entry shall include an `applied_at` ISO 8601 timestamp recording when it was last written by `apply()`.                                                                   | Must Have   | PP-5, PP-6, G-1 |
| FR-REG-002 | The `apply()` operation shall scan the `definitions/` directory, discover all `FeatureGroup` instances, validate them, and fully regenerate the registry. If any definition is invalid, the registry remains unchanged and all errors are reported. The registry is a derived artifact rebuilt entirely each time from source definitions. However, runtime-managed fields (`last_materialized_at`) are preserved across `apply()` runs by reading them from the existing registry before overwriting. This is a project-level operation exposed via CLI — not intended for programmatic use in notebooks or scripts. | Must Have   | G-1, PP-6       |
| FR-REG-003 | Feature group discovery during `apply()` shall use `isinstance` checks on module-level attributes — no decorators, naming conventions, or registration calls required. Defining a `FeatureGroup` object in a file under `definitions/` is sufficient.                                                                                                                   | Must Have   | G-1             |
| FR-REG-004 | The system shall enforce that feature group names are unique within a project. If a duplicate name is found during `apply()`, the operation shall fail with a clear error.                                                                                                                                                                                               | Must Have   | G-1, PP-6       |
| FR-REG-005 | The user shall be able to list all registered feature groups via the SDK and the CLI, seeing at minimum: name, owner, entity key, storage target, and feature count.                                                                                                                                                                                                     | Must Have   | PP-5            |
| FR-REG-006 | The user shall be able to retrieve the full definition of a specific feature group (including all features, types, expectations, join keys, and validation modes) by name via the SDK and the CLI.                                                                                                                                                                       | Must Have   | PP-5            |
| FR-REG-007 | The registry shall store the relationship metadata needed for cross-feature-group joins — specifically, which fields in one feature group can be used to join with the entity key of another feature group.                                                                                                                                                              | Must Have   | G-2             |
| FR-REG-008 | The user shall be able to sync the local registry file to a remote storage location (S3 in the AWS provider) via the CLI.                                                                                                                                                                                                                                               | Should Have | PP-5            |
| FR-REG-009 | The user shall be able to pull a remote registry file to the local environment via the CLI, overwriting the local copy.                                                                                                                                                                                                                                                  | Should Have | PP-5            |

### 1.3 Data Ingestion

| ID         | Requirement                                                                                                                                                                                                                                                                 | Priority  | Traces To      |
| ---------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------- | -------------- |
| FR-ING-001 | The user shall be able to ingest feature data into the offline store for a registered feature group by providing a Pandas DataFrame, a local CSV file path, or a local Parquet file path, together with the target feature group name, via the SDK. DataFrame ingestion is available only through the SDK.                         | Must Have | G-1, G-3, G-4  |
| FR-ING-002 | The system shall validate that the ingested data's columns match the feature group's schema (field names and data types) before writing. Columns present in the DataFrame but absent from the definition are silently dropped. Columns defined in the feature group but missing from the DataFrame cause a rejection with a clear error.                                                                                   | Must Have | G-5, PP-7      |
| FR-ING-003 | The system shall run the validation engine on ingested data at the ingestion gate according to the feature group's configured ingestion validation mode (`ERROR`, `FILTER`, or `NONE`). Mode behaviors are defined in FR-VAL-004 through FR-VAL-006.                        | Must Have | G-5, PP-7      |
| FR-ING-004 | Ingestion shall be append-only. Each operation creates new Parquet file(s) for the affected partition(s) without overwriting or modifying previously ingested data.                                                                                                         | Must Have | G-1            |
| FR-ING-005 | Every ingested record must include a valid `event_timestamp` value. Records without an `event_timestamp` shall be rejected.                                                                                                                                                 | Must Have | G-2            |
| FR-ING-006 | Ingested data shall be written as Parquet files using a Hive-style partitioned directory layout: `feature_group_name/year=YYYY/month=MM/`. This layout is identical for both local filesystem and S3 and supports partition pruning for efficient time-range queries.        | Must Have | G-3, G-4       |
| FR-ING-007 | Each Parquet file written to the offline store shall follow the naming pattern `{source}_{write_timestamp}_{short_id}.parquet`, where: `source` is a short prefix identifying the write origin (e.g., `ing` for ingestion), `write_timestamp` is the UTC time of the write operation in filesystem-safe ISO format (`YYYYMMDDTHHMMSS`), and `short_id` is a short random alphanumeric string for collision avoidance. The source prefix is informational only — the system shall treat all `.parquet` files within a partition identically regardless of prefix. | Must Have | G-3, G-4       |

### 1.4 Offline Store & Historical Retrieval

| ID         | Requirement                                                                                                                                                                                                                                                                                                                                                                                                                          | Priority   | Traces To       |
| ---------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ---------- | --------------- |
| FR-OFF-001 | The system shall store offline feature data as Parquet files — on the local filesystem (local provider) or S3 (AWS provider) — organized by feature group name and partitioned by `year=YYYY/month=MM`. Both providers shall be accessible through the same SDK methods.                                                                                                                                                             | Must Have  | G-3, G-4        |
| FR-OFF-002 | The `get_historical_features` SDK method shall accept: `from_=` (base feature group — its rows define the output), `select=` (features to retrieve), optional `where=` (row filter on the base group using the unified where format — see FR-OFF-007), and optional `join=` (feature groups to join). It returns a Pandas DataFrame including the base group's entity key and `event_timestamp` as structural columns. Without `join`, `select` may be a list of field names or `"*"` (all fields: entity key, event timestamp, and all features). With `join`, `select` must be a dict keyed by feature group name; `"*"` is valid as a dict value for any group. For the `from_` group, `"*"` returns: entity key, event timestamp, and all features (including join key fields). For a joined group, `"*"` returns: all features plus `event_timestamp`; the joined group's entity key is not separately included because it is the join key already present from the `from_` group. Column name conflicts between the `from_` group and joined groups are resolved per FR-OFF-010. When no `join` is provided, the system reads the base group, applies any `where` filter, and returns the result — no join or point-in-time logic is applied. | Must Have  | G-2, PP-1       |
| FR-OFF-003 | When joining across feature groups, the system shall enforce point-in-time correctness: for each base record, join only the most recent right-side record whose `event_timestamp` is less than or equal to the base record's `event_timestamp`. This prevents data leakage.                                                                                                                                                          | Must Have  | G-2, PP-4, PP-1 |
| FR-OFF-004 | The user shall be able to declare join keys in a feature group definition. Each join key specifies a field in the current group and references another registered feature group. At query time, the declared field is joined to the referenced group's entity key. Join key declarations are validated at registration time: the field must exist, the referenced group must be registered, and the field type must match the referenced entity key type. The join key field name must match the entity key name of the referenced group. Mismatched names shall be rejected during `apply()` with a clear error. | Must Have  | G-2             |
| FR-OFF-005 | When a point-in-time join cannot find a matching right-side record, the joined columns shall be filled with null values. The base record shall not be dropped.                                                                                                                                                                                                                                                                       | Must Have  | G-2             |
| FR-OFF-006 | The system shall run the validation engine on offline store data before performing joins. Each feature group is validated independently per its offline retrieval validation mode. In `ERROR` mode, failure in any group rejects the entire operation. In `FILTER` mode, only passing records proceed to the join.                                                                                                                             | Must Have  | G-5, PP-7       |
| FR-OFF-007 | The `where` parameter uses a unified format across all SDK retrieval methods: `where={field_name: {operator: value}, ...}`. The full operator set is: `eq` (equals), `in` (in a list), `gt` (greater than), `gte` (greater than or equal), `lt` (less than), `lte` (less than or equal). Multiple operators on the same field combine with AND semantics. **MVP restriction for `get_historical_features`:** only `event_timestamp` is accepted as a field name, and only `gt`, `gte`, `lt`, `lte` operators are accepted. The `where` filter applies exclusively to the base feature group (`from_`); joined groups are not filtered by `where` — their temporal scope is determined by the point-in-time join condition. Example: `where={"event_timestamp": {"gte": datetime(...), "lte": datetime(...)}}`. Datetime comparisons are timestamp-precise (including time component). The unified format is designed to support arbitrary field filters in the future by relaxing the per-method validation rules — no API signature change required. | Must Have  | G-2, PP-1       |
| FR-OFF-008 | When `join` is provided, the system shall resolve joins using relationship metadata from feature group definitions (join keys). If no valid join path exists or a referenced group is not registered, the system shall raise a clear error.                                                                                                                                                                                           | Must Have  | G-2             |
| FR-OFF-009 | **MVP restriction:** The system shall enforce a single-join limit: the `join` parameter accepts at most one feature group. If more than one is provided, the system shall reject the call with a clear error. This validation is isolated in a single check, making it straightforward to relax when multi-join support is added. Multi-join (joining the base with multiple groups in a single call) is deferred as a future enhancement beyond the MVP.                                                                                                                                                                                                                                                                                   | Must Have  | G-2, PP-1       |
| FR-OFF-010 | When a point-in-time join produces column name conflicts between the base (`from_`) group and a joined group, the system shall resolve conflicts by prefixing the conflicting joined column with `{joined_group_name}_` (e.g., a conflicting `event_timestamp` from `town_market_features` becomes `town_market_features_event_timestamp`). Base group columns are never renamed. Non-conflicting joined columns keep their original names. The joined group's `event_timestamp` is always included in the output as a structural column and is always prefixed — since it invariably conflicts with the base group's `event_timestamp`. The join key column is not duplicated; it appears once from the base group. | Must Have  | G-2             |

### 1.5 Online Store & Feature Serving

| ID         | Requirement                                                                                                                                                                                                                                                                                                                                                                                                                                                                              | Priority   | Traces To |
| ---------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------- | --------- |
| FR-ONL-001 | The system shall store online feature data in SQLite (local provider) or DynamoDB (AWS provider). Both shall be accessible through the same SDK methods.                                                                                                                                                                                                                                                                                                                                | Must Have  | G-3, G-4  |
| FR-ONL-002 | The user shall be able to retrieve the latest feature values for a single entity key from the online store via `get_online_features(from_, select, where)`. <br><br> **Parameters:** `from_` specifies the feature group (must be `OFFLINE_AND_ONLINE`). `select` specifies feature names (list or `"*"`). `where` uses the unified where format (see FR-OFF-007) with the entity key as the filter field (e.g., `where={"town_id": {"eq": 1}}`). <br><br> **Returns:** `dict \| None` — a dict of feature values (including structural columns) if the entity key exists, or `None` if it does not. Example: `{"town_id": 1, "event_timestamp": "2025-06-01T00:00:00", "avg_price_per_sqm": 27800.0}` for a found key; `None` for a missing key. <br><br> **Select behavior:** When `select="*"`, all declared features are returned alongside the structural columns (entity key and `event_timestamp`). When `select` is a list, only the named features are returned alongside the structural columns. Structural columns are always included regardless of `select`. <br><br> **Behavioral notes:** The online store holds only the latest values per entity key; previous values are overwritten on materialization. If the feature group's online store table does not exist (i.e., `materialize()` has never been run), the system shall return a clear error indicating that materialization is required. <br><br> **MVP restriction:** Only the entity key field is accepted as a `where` field name, only the `eq` operator is accepted, and only a single value is allowed. The system resolves which field is the entity key from the feature group definition. | Must Have  | G-1, PP-3 |
| FR-ONL-003 | The system could support multiple entity keys in a single `get_online_features` call (e.g., `where={"town_id": {"in": [1, 2, 3]}}`). The unified where format already includes the `in` operator; enabling batch support requires relaxing the MVP validation restriction from `eq`-only to also accept `in` — no API signature change required. The return type would change from `dict \| None` (single-key MVP) to `list[dict \| None]` (one element per entity key in input order; `None` for missing keys). Deferred beyond MVP. | Could Have | PP-3      |

### 1.6 Materialization

| ID         | Requirement                                                                                                                                                                                                                                                                                                                                      | Priority    | Traces To |
| ---------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ----------- | --------- |
| FR-MAT-001 | The user shall be able to trigger materialization for a feature group configured as `OFFLINE_AND_ONLINE` via the SDK and the CLI. Materialization reads the latest value per entity key from the offline store and writes it to the online store (full overwrite per group). Without a feature group argument, all eligible groups are materialized independently — a failure in one group does not prevent other groups from being materialized. If a group's offline store is empty, it is skipped with a warning. The system records a `last_materialized_at` ISO 8601 timestamp in the registry JSON for each successfully materialized group. | Must Have   | G-1, PP-3 |
| FR-MAT-002 | If materialization is triggered for a feature group configured as `OFFLINE` only, the system shall reject the operation with a clear error.                                                                                                                                                                                                      | Must Have   | G-1       |
| FR-MAT-003 | Materialization shall be idempotent: running it multiple times with the same offline data produces the same online store state.                                                                                                                                                                                                                   | Must Have   | G-1       |
| FR-MAT-004 | Materialization shall be an explicit, user-triggered operation. The system shall not schedule or auto-trigger materialization.                                                                                                                                                                                                                    | Must Have   | NG-1      |
| FR-MAT-005 | After materialization completes, the system shall report how many entity keys were written to the online store.                                                                                                                                                                                                                                  | Must Have   | G-5       |
| FR-MAT-006 | The user shall be able to specify a time range for materialization to support incremental updates.                                                                                                                                                                                                                                                | Could Have  | G-1       |

### 1.7 Validation Engine

| ID         | Requirement                                                                                                                                                                                                                                                                                         | Priority  | Traces To |
| ---------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------- | --------- |
| FR-VAL-001 | The validation engine shall perform type validation: checking that each feature value conforms to the declared data type (`string`, `integer`, `float`, `datetime`).                                                                                                                                | Must Have | G-5, PP-7 |
| FR-VAL-002 | The validation engine shall enforce feature expectations: checking each feature value against user-defined business constraints (minimum value, maximum value, allowed value set, not-null).                                                                                                        | Must Have | G-5, PP-7 |
| FR-VAL-003 | The system shall support three validation modes — `ERROR`, `FILTER`, and `NONE` — configurable independently at each validation gate (ingestion, offline retrieval) per feature group. Type validation and feature expectations are enforced together under the active mode; they are not independently toggleable. | Must Have | G-5       |
| FR-VAL-004 | In `ERROR` mode, if any record fails validation, the entire operation shall be rejected (no data written or returned) with an actionable error report.                                                                                                                                             | Must Have | G-5, PP-7 |
| FR-VAL-005 | In `FILTER` mode, passing records proceed normally; failing records are excluded. The system shall produce a report identifying excluded records and reasons.                                                                                                                                       | Must Have | G-5, PP-7 |
| FR-VAL-006 | In `NONE` mode, validation is skipped entirely.                                                                                                                                                                                                                                                     | Must Have | G-5       |
| FR-VAL-007 | Validation reports (in `ERROR` and `FILTER` modes) shall include: passed record count, failed record count, and for each failure the entity key value (where available), failing field(s), expected constraint, and actual value.                                                                   | Must Have | G-5, PP-7 |
| FR-VAL-008 | The validation engine shall support at minimum: `gt`, `gte`, `lt`, `lte`, `one_of` (value in a specified set), and `not_null`.                                                                                                                                                                     | Must Have | G-5, PP-7 |
| FR-VAL-009 | When validation modes are not explicitly set, the system shall default to `ERROR` for ingestion and `NONE` for offline retrieval.                                                                                                                                                                           | Must Have | G-5       |

## 2. Non-Functional Requirements

> _Essential quality attributes that cut across all features. Kite is a
> library, not a hosted service — these focus on data integrity, developer
> experience, and code quality, not uptime SLAs._

### 2.1 Data Integrity

| ID          | Requirement                                                                                                                                                                                                                           | Target                             |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------- |
| NFR-REL-001 | The system shall never write partial data to the offline store on ingestion failure. If validation fails in `ERROR` mode or an I/O error occurs, the offline store shall remain in its pre-ingestion state.                           | Zero partial writes on any failure |
| NFR-REL-002 | The system shall never write partial data to the online store on materialization failure.                                                                                                                                              | Zero partial writes on failure     |
| NFR-REL-003 | Point-in-time join results shall be deterministic: given the same offline data and query parameters, `get_historical_features` shall always return the same result.                                                                   | 100% deterministic                 |

### 2.2 Usability

| ID         | Requirement                                                                                                                                                                                                                                                |
| ---------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| NFR-UX-001 | Error messages from the SDK and CLI shall be actionable: state what went wrong, which input caused the error, and what the user can do to fix it. Generic messages like "An error occurred" are unacceptable.                                              |
| NFR-UX-002 | The SDK shall accept and return standard Python data structures: Pandas DataFrames for batch operations (ingestion, historical retrieval) and `dict \| None` for online retrieval (single-key MVP; future batch support would return `list[dict \| None]`). No custom container types required for basic usage. |

### 2.3 Maintainability

| ID            | Requirement                                                                                                                                                                                                                                   |
| ------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| NFR-MAINT-001 | The codebase shall follow a modular architecture with separation between: core logic (definitions, validation, joins), provider interface, provider implementations, SDK/public API, and CLI. No circular dependencies between modules.        |
| NFR-MAINT-002 | The provider abstraction shall be enforced via a clearly defined interface (Python abstract base class or Protocol). Adding a new provider requires only implementing this interface — no changes to core logic.                               |

### 2.4 Security

| ID          | Requirement                                                                                                                                                                                                          |
| ----------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| NFR-SEC-001 | When using the AWS provider, the system shall rely on the standard AWS credential chain and IAM policies for S3 and DynamoDB access. The library shall not store, manage, or transmit AWS credentials itself.        |

### 2.5 Constraints

> _Hard constraints that limit design choices. Non-negotiable boundaries._

| ID      | Constraint                                                                                                                                                           | Rationale                                                                                     |
| ------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------- |
| CON-001 | The library shall be implemented in Python 3.12 or higher.                                                                                                           | Modern Python features; aligns with target user base.                                         |
| CON-002 | The library shall be distributed as a single pip-installable package. No companion services or infrastructure provisioning required for the local provider.          | Core to the library-first philosophy (G-3).                                                   |
| CON-003 | The library shall use Pandas as the primary DataFrame interface for ingestion and retrieval.                                                                         | Most widely adopted DataFrame library in the Python data science ecosystem.                   |
| CON-004 | The MVP shall support only a single entity key per feature group. Composite keys are out of scope.                                                                  | Avoids complexity in joins, online-key structure, and retrieval APIs.                         |
| CON-005 | The system shall not depend on any running server process or daemon. All operations are library calls or CLI commands.                                               | Simplicity and low operational burden for the MVP.                                             |
| CON-006 | Scope is limited to core feature store capabilities. Enterprise features (streaming ingestion, web UI, multi-cloud, drift detection, custom auth) are out of scope. | Aligns with Non-Goals (NG-1 through NG-7) in the Project Charter.                            |

## 3. Assumptions

> _Things we believe to be true that, if wrong, would change requirements._

| ID    | Assumption                                                                                                                                                                                    | Impact if Wrong                                                                                               |
| ----- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------- |
| A-001 | Users have Python 3.12+ installed.                                                                                                                                                            | Would need to lower minimum version, losing modern language features.                                         |
| A-002 | AWS provider users have an AWS account with appropriate IAM permissions for S3 and DynamoDB.                                                                                                 | AWS provider would fail. Need detailed setup docs or permissions-check utility.                               |
| A-003 | The primary ingestion format is Pandas DataFrames. Users can convert from other formats (Polars, PySpark, CSV, SQL results) to Pandas.                                                       | Would need native support for additional DataFrame libraries.                                                 |
| A-004 | Feature groups are small to medium width (up to ~100 features). The reference use case has 5–7 features per group.                                                                           | Wide groups could hit SQLite schema limits or DynamoDB item size limits.                                      |
| A-005 | Offline store volume per feature group stays in the low single-digit millions. The reference use case has ~2M listing records and ~72 market records.                                         | Significantly larger volumes (100M+) would require a query engine (e.g., DuckDB) or advanced partitioning.    |
| A-006 | AWS network latency is not a bottleneck. Online serving assumes same-region deployment.                                                                                                      | Cross-region usage would not meet latency expectations. Need region documentation or caching.                |
| A-007 | The MVP targets single-user, single-process usage. Concurrent writes to the same feature group are not supported.                                                                            | Would need file locks (local) or conditional writes (DynamoDB) to prevent corruption.                        |
| A-008 | Point-in-time join uses a less-than-or-equal-to comparison (joined record's `event_timestamp` may equal the source record's).                                                                | Strict "before only" semantics would need a different join rule or timestamping convention.                   |
| A-009 | Event timestamps are user-provided and semantically meaningful. The system does not generate or infer them.                                                                                   | Incorrect timestamps would produce wrong join results. Validation catches type/range, not semantic meaning.  |
